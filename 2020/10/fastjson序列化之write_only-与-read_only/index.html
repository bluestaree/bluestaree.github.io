fastjson序列化之WRITE_ONLY 与 READ_ONLY

  2020-10-05   笔记   json  评论 字数统计: 481字 阅读时长: 1分

前言

在项目开发中，经常会遇到这样的场景：对于参数较多的接口，一般是定义参数VO类的方式进行接收，同时根据需要定义返回的VO类。这样做项目结构清晰，不存在多余属性值，缺点是：VO类定义多。另外的做法是，尽量将入参和返回信息都定义在都一个VO类中。缺点是前端会接收到无关的属性信息，影响前端开发。

本文主要接收fastjson序列化参数之 WRITE_ONLY 与 READ_ONLY 的配置

此配置也可以解决 使用@JsonIgnore 属性后无法接收前端参数的问题

当接口中请求参数Bean的字段被 @JsonProperty(access = Access.WRITE_ONLY) 约束时，能够正常接收参数（反序列化），但不参与序列化

反之，被 @JsonProperty(access = Access.READ_ONLY) 约束时的属性，不能接收到前端传递的参数，能正常进行序列化，但不参与反序列化

测试准备

实体类

@Data
@ToString
public class User implements Serializable {

    private static final long serialVersionUID = 1L;

    @TableId(value = "id", type = IdType.AUTO)
    private Long id;

    @JsonProperty(access = JsonProperty.Access.READ_ONLY)
    private String username;

    @JsonProperty(access = JsonProperty.Access.READ_ONLY)
    private String avatar;

    @JsonProperty(access = JsonProperty.Access.WRITE_ONLY)
    private String email;

    @JsonProperty(access = JsonProperty.Access.WRITE_ONLY)
    private String password;

}

控制层

@Slf4j
@RestController
@RequestMapping("/test")
public class JsonController {

    @PostMapping("/json")
    public User getUser(@RequestBody User user) {
        log.info("接收到的参数：{}" ,user);
        user.setId(1L);
        user.setUsername("username1");
        user.setAvatar("avatar1");
        user.setEmail("email1");
        return user;
    }

}
